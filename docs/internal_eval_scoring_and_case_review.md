# 자체 평가 데이터 자동채점 기준 및 정성 검산 위치

작성일: 2026-06-16

이 문서는 보고서의 `자체 평가 데이터 자동채점 결과` 그림에 사용된 평가 기준, 실제 산출물, 결과표, 정성 검산 자료 위치를 공개하기 위한 보조 문서이다.

## 1. 결론

- 보고서 그림의 결과는 실제 실행 결과에서 나온 값이다.
- 채점자는 사람이 아니라 로컬 규칙 기반 자동채점기이다.
- 제한 평가에 사용한 원래 자동채점 스크립트는 `scripts/evaluate_target_posttrain_tasks.py`이다.
- 전체 803문항 재평가에 사용한 엄밀화 스크립트는 `scripts/evaluate_target_posttrain_tasks_strict.py`이다.
- 이 평가는 KMMLU, CLIcK, KoBEST 같은 공개 객관식 벤치마크가 아니라, 프로젝트 내부 목표 작업 평가셋에 대한 생성형 답변 진단이다.
- 사람이 점수를 매긴 별도 정량 평가는 없다.
- 사람이 점수를 매긴 평가지표표는 확인되지 않으며, 확인 가능한 것은 자동채점 후보 사례를 보고서에 실어도 되는지 대조한 정성 검산 자료이다.

## 4. 기존 제한 평가 자동채점 기준

이 절은 보고서 초안의 제한 평가, 즉 `limit_per_task=20` 조건으로 실행한 139개 생성 결과에 적용된 원래 자동채점 기준이다. 자동채점은 정답 문장 전체를 사람이 읽고 점수화한 방식이 아니다. 각 문항에 들어 있는 `required_sections`, `required_terms`, `forbidden_terms`, `answer_key`를 기준으로 답변을 규칙적으로 채점한다.

다만 이 원래 기준에는 요구 구획이 없는 문항에 구획 점수가 자동으로 들어가고, 단답형 문항에도 20자 이상 길이 조건을 적용하는 문제가 있었다. 전체 803문항 재평가에서는 이 부분을 9.1~9.2절처럼 수정했다.

### 텍스트형 문항

텍스트형 문항의 최종 점수는 다음 가중합이다.

`최종 점수 = 0.35 * 섹션 충족률 + 0.25 * 필수어 충족률 + 0.25 * 금지어 회피 + 0.15 * 길이 조건`

| 구성 요소 | 의미 |
|---|---|
| 섹션 충족률 | 요구된 답변 구획이 출력에 들어 있는 비율이다. 예: 안건, 주요 발언, 쟁점, 근거, 한계/추가 확인 |
| 필수어 충족률 | 문항별 필수 표현이나 핵심 용어가 출력에 들어 있는 비율이다 |
| 금지어 회피 | 문항에서 금지한 표현이나 자료 밖 단정이 없으면 1점, 있으면 0점이다 |
| 길이 조건 | 답변 길이가 20자 이상 2500자 이하이면 1점, 아니면 0점이다 |

### 수치형 문항

수치형 문항의 최종 점수는 다음 가중합이다.

`최종 점수 = 0.45 * 수치 정답 + 0.15 * 단위 포함 + 0.15 * 필수어 충족률 + 0.15 * 금지어 회피 + 0.10 * 길이 조건`

| 구성 요소 | 의미 |
|---|---|
| 수치 정답 | 출력에서 숫자를 추출한 뒤 문항의 `answer_key` 값과 허용 오차 안에 있으면 1점이다 |
| 단위 포함 | 문항이 요구한 단위가 출력에 포함되어 있으면 1점이다. 예: `%p`, `억 원` |
| 필수어 충족률 | 문항별 필수 표현이나 핵심 용어가 출력에 들어 있는 비율이다 |
| 금지어 회피 | 문항에서 금지한 표현이나 자료 밖 단정이 없으면 1점, 있으면 0점이다 |
| 길이 조건 | 답변 길이가 20자 이상 2500자 이하이면 1점, 아니면 0점이다 |

## 6. 해석

지도학습(SFT)은 단답, 기준일이 있는 사실 질문, 기본 정책 설명처럼 답변 형식이 뚜렷한 문항에서 개선 폭이 컸다. 이는 사전학습 베이스라인과 추가 사전학습(CPT)에서 나타난 장황한 반복, 질문과 무관한 법률 문구 생성, 근거 없는 일반론을 줄이는 데 SFT가 직접적으로 작용했기 때문으로 볼 수 있다.

선호학습(DPO)은 전체 원시 평균에서 SFT보다 소폭 높았고, 보고서 묶음 기준으로는 회의록 구조화, 법률안·검토보고서 해설, 근거 부족·짧은 문서 진단, 표 수치 해석에서 SFT보다 높았다. 이는 DPO가 실패 답변 교정, 자료 밖 단정 억제, 반복 억제 같은 선호 기준을 강화한 결과로 해석할 수 있다.

다만 공약·정책 비교와 기본 정치 사실·단답에서는 DPO가 SFT보다 낮았다. 이는 DPO 보강쌍이 모든 목표 작업을 균등하게 끌어올린 것이 아니라, 반복 억제와 실패사례 교정에 더 집중되었기 때문으로 추정된다. 또한 보고서 그림의 항목별 문항 수가 15~27개 수준이므로, 일부 항목의 작은 하락을 일반 성능 하락으로 단정하기보다는 후속 실험에서 데이터 균형과 선호쌍 구성을 조정할 개선점으로 보는 편이 안전하다.

## 7. 사람이 평가한 정성평가는 어디에 있는가

사람이 점수를 매긴 독립 정량 평가는 따로 없다. 즉, 보고서 그림의 점수는 사람 평가 점수가 아니다. 사람이 별도 평가지표표를 작성해 채점한 파일도 확인되지 않는다.

대신 확인 가능한 것은 보고서 작성 과정에서 수행한 정성 검산 자료이다. 이는 점수표가 아니라, 자동채점 후보가 실제 개선 사례로 볼 수 있는지 원문과 출력을 대조한 검토 기록이다.

첫째, 자동채점 결과로 뽑힌 후보가 실제로 같은 prompt를 네 단계에 통과시켜 얻은 사례인지 기계적으로 검산했다. 이 검산에서는 prompt 정렬, train split 포함 여부, completion 불일치 여부, 단계별 pass vector를 확인했다.

검산 파일:

`examples/improvement_cases/cumulative_transition_case_review.md`

둘째, 보고서에 넣을 사례가 의미상 과장되지 않았는지 정성 검산했다. 이 과정에서 일부 사례는 자동 pass였더라도 오탈자, 어색한 대응, 기준답변과의 미묘한 불일치 때문에 제외하거나 후순위로 두었다.

사례 보고서:

`examples/improvement_cases/cumulative_transition_cases.md`

확장 사례 후보:

`examples/improvement_cases/selected_improvement_cases_expanded.md`

따라서 보고서에는 다음처럼 쓰는 것이 정확하다.

> 자체 평가 데이터는 프로젝트 목표 작업에 맞춘 내부 생성형 진단셋이며, 평가는 규칙 기반 자동채점으로 수행하였다. 별도 사람 점수 평가는 수행하지 않았고, 보고서에 수록한 개선 사례는 자동채점 후보 중 원문, 질문, 단계별 출력을 대조하여 정성 검산한 사례이다.

## 9. 전체 803문항 재평가 결과

2026-06-16 10:12~10:23 UTC에 제한 평가가 아니라 전체 target_eval 803문항을 네 단계 모두에 대해 다시 생성·채점했다. 이 실행은 기존 `limit_per_task=20` 제한 평가와 구분하기 위해 `full803_strict` 이름으로 저장했다.

이번 재평가에서는 `target_eval_strict_v2` 채점 기준을 적용했다. 주요 차이는 다음과 같다.

- 요구 구획이 없는 문항에는 구획 점수 35점이 자동으로 들어가지 않는다.
- 단답형 또는 기준답변이 짧은 문항은 20자 미만이라는 이유만으로 감점하지 않는다.
- 필수어와 금지어는 유니코드 정규화와 공백 정규화를 거쳐 대조한다.
- 수치형 문항은 정답 수치뿐 아니라 가능한 경우 방향 또는 입력값 언급도 보조적으로 확인한다.

### 9.1 기존 제한 평가에서 바로잡은 점

기존 제한 평가는 보고서 초안 작성 중 빠르게 단계별 경향을 확인하기 위해 문항 유형별 최대 20개만 뽑아 실행한 진단이었다. 이때 사용한 원래 자동채점식은 전체 추세를 보는 데는 사용할 수 있었지만, 보고서에 자체 평가 기준으로 공개하기에는 다음 문제가 있었다.

첫째, 요구 구획이 없는 문항도 `section_score=1.0`으로 처리했다. 따라서 단답, 개념 설명, 현재 사실 확인처럼 애초에 구획을 요구하지 않은 문항이 구획 점수 35점을 자동으로 받았다. 이 처리는 실제 답변 품질보다 점수를 높게 보이게 할 수 있었다.

둘째, 모든 문항에 같은 길이 조건을 적용해 답변 길이가 20자 미만이면 감점했다. 그러나 "서울입니다."처럼 단답으로 끝나는 것이 맞는 문항은 짧은 답변 자체가 정상 형식이다.

셋째, 전체 후보 문항 803개가 있는데도 보고서 초안에는 `limit_per_task=20` 제한 평가에서 나온 139개 생성 결과와, 그중 6개 항목으로 묶은 123개 결과가 함께 설명되어 혼동을 줄 수 있었다.

이번 `target_eval_strict_v2`에서는 위 문제를 다음처럼 수정했다.

| 수정 전 | 수정 후 |
|---|---|
| 문항에 요구 구획이 없어도 구획 점수를 1점으로 처리 | 요구 구획이 없는 문항은 구획 항목을 계산에서 제외하고 나머지 구성 요소를 정규화 |
| 모든 문항에 20~2,500자 길이 조건 적용 | 단답형 또는 기준답변이 짧은 문항은 2~800자 조건 적용 |
| 139개 제한 평가를 보고서용 대표값처럼 설명 | 803개 전체 평가를 별도로 실행하고 `full803_strict` 결과로 분리 |
| 공백 차이와 일부 표기 차이에 취약한 문자열 대조 | 유니코드 정규화와 공백 정규화 후 필수어·금지어 대조 |

### 9.2 전체 803문항 자동채점 수식

점수는 0~1 범위로 계산한 뒤 100을 곱해 표시했다. `target_eval_strict_v2`에서는 해당 문항에 실제로 적용되는 구성 요소만 사용하고, 빠진 구성 요소의 가중치는 남은 구성 요소 안에서 다시 정규화한다.

텍스트형 문항의 기본 수식은 다음과 같다.

`텍스트형 점수 = 정규화(0.35 × 요구 구획 충족률 + 0.25 × 필수어 충족률 + 0.25 × 금지어 회피 + 0.15 × 길이 조건)`

수치형 문항의 기본 수식은 다음과 같다.

`수치형 점수 = 정규화(0.45 × 수치 정답 + 0.15 × 단위 포함 + 0.15 × 필수어 충족률 + 0.15 × 금지어 회피 + 0.10 × 길이 조건)`

구성 요소의 의미는 다음과 같다.

| 구성 요소 | 적용 문항 | 의미 |
|---|---|---|
| 요구 구획 충족률 | 텍스트형 중 요구 구획이 있는 문항 | 안건/문맥, 쟁점, 근거, 한계처럼 문항이 요구한 답변 구획이 포함된 비율 |
| 필수어 충족률 | 텍스트형·수치형 | 문항별 핵심 용어 또는 필수 표현이 포함된 비율 |
| 금지어 회피 | 텍스트형·수치형 | 금지 표현이나 자료 밖 단정이 없으면 1점, 있으면 0점 |
| 길이 조건 | 텍스트형·수치형 | 일반 문항은 20~2,500자, 단답형 문항은 2~800자이면 1점 |
| 수치 정답 | 수치형 | 출력의 숫자가 정답값과 허용 오차 안에 있으면 1점 |
| 단위 포함 | 수치형 | `%p`, `억 원` 등 문항이 요구한 단위가 포함되면 1점 |
